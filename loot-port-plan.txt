# Plan: Port Loot System from NexSuite2 NexAi to NexCore NexAi

## Context

Porting the Loot subsystem (LootManager + VTankLootParser) from the old Decal-dependent NexAi at C:\Projects\NexSuite2\Plugins\NexAi\Loot\ to the new NexCore-based NexAi at C:\Projects\NexSuite\Plugins\NexCore.Plugin.NexAi\. The goal is to copy the files and adapt them to use NexCore's API instead of Decal's.

### Decisions
- Salvage: Implement now — find VAs for SalvagePanelAdd/SalvagePanelSalvage in acclient.exe
- Loot evaluation: VTankLootParser IS the reimplemented VTank loot engine (handles UTL rule trees). Remove VTClassic/uTank2 code — our parser is the sole evaluator. Can add more VTank node types later if profiles need them.
- NeedsID: Always call RequestId on corpse items, wait up to assess window

## Must Add to NexCore Engine + SDK

### 1. InqString — String Property Reading
File: src/NexCore.Engine/Compatibility/ClientObjectHooks.cs

Why: VTank rule type 1 (StringValueMatch) reads string properties. Corpse ownership reads StringValueKey 16 (LongDesc = "Killed by {name}.").

- Find VA for CBaseQualities::InqString (near InqInt 0x00590C20 / InqBool 0x00590CA0)
- Add delegate: int InqString(IntPtr qualitiesPtr, uint stype, IntPtr* retval) (ThisCall)
- Add method: TryGetObjectStringProperty(uint objectId, uint stype, out string value)
- Wire through NexCoreApiNative.GetObjectStringPropertyFn -> NexCoreHost.TryGetObjectStringProperty()

### 2. InqFloat — Float/Double Property Reading
File: src/NexCore.Engine/Compatibility/ClientObjectHooks.cs

Why: VTank rule types 4 & 5 (DoubleValKeyLE/GE) compare double properties (weapon damage variance, attack mod, etc.).

- Find VA for CBaseQualities::InqFloat (near InqInt/InqBool)
- Add delegate + method: TryGetObjectFloatProperty(uint objectId, uint stype, out double value)
- Wire through SDK

### 3. Container Contents API
File: src/NexCore.Engine/Compatibility/ClientObjectHooks.cs or UpdateObjectInventoryHooks.cs

Why: After opening a corpse, need to enumerate items inside via GetByContainer(corpseId).

- Expose existing GetExhaustiveContainedItemsList as SDK function: GetContainerContents(uint containerId, uint* outIds, int maxCount) -> int
- Wire through NexCoreApiNative.GetContainerContentsFn -> NexCoreHost.GetContainerContents()

### 4. Salvage Panel APIs
File: New src/NexCore.Engine/Compatibility/SalvagePanelHooks.cs (or add to ClientHelperHooks)

Why: Auto-salvage system needs SalvagePanelAdd(itemId) and SalvagePanelSalvage().

- Find VAs in acclient.exe APIManager__IAsheronsCallImpl_vtbl (documented as "declared but not yet validated")
- Implement as direct function calls via delegates
- Wire through SDK

### 5. ObjectClass Expansion
File: Plugins/NexCore.Plugin.NexAi/Combat/AcStubs.cs + WorldObjectCache.cs

Why: Loot system checks ObjectClass.Corpse, .Salvage, .Container, .Foci.

- Expand AcObjectClass enum with Corpse, Container, Salvage, Foci values
- Add ITEM_TYPE constants: TYPE_CORPSE=0x00000020, TYPE_CONTAINER=0x00000040
- Update WorldObjectCache.ClassifyByItemType() to detect these types

### 6. String + Double Property Access on WorldObject
File: Plugins/NexCore.Plugin.NexAi/Combat/AcStubs.cs

- Add StringValueKey enum (at minimum key 16 = LongDesc)
- Add DoubleValueKey enum
- Add WorldObject.Values(StringValueKey, default) -> delegates to cache -> SDK
- Add WorldObject.Values(DoubleValueKey, default) -> delegates to cache -> SDK
- Add WorldObjectCache.GetStringProperty() and WorldObjectCache.GetFloatProperty() methods

## Implementation Order

### Phase 1: Engine — Property APIs
1. Find VAs for CBaseQualities::InqString and CBaseQualities::InqFloat
2. Add to ClientObjectHooks.cs: delegates, TryGetObjectStringProperty(), TryGetObjectFloatProperty()
3. Add to NexCoreApiNative.cs: GetObjectStringPropertyFn, GetObjectFloatPropertyFn
4. Add to NexCoreHost.cs: wrapper methods
5. Wire function pointers in EntryPoint.cs

### Phase 2: Engine — Container Contents API
1. Add GetContainerContents() wrapper around existing GetExhaustiveContainedItemsList
2. Wire through SDK

### Phase 3: Engine — Salvage Panel APIs
1. Find VAs for salvage panel functions in acclient.exe vtable
2. Implement + wire through SDK

### Phase 4: Plugin — Expand Type System
1. AcStubs.cs: Add StringValueKey, DoubleValueKey enums
2. AcStubs.cs: Expand AcObjectClass (Corpse, Container, Salvage, Foci)
3. WorldObject: Add Values(StringValueKey), Values(DoubleValueKey) methods
4. WorldObjectCache: Add GetStringProperty(), GetFloatProperty(), expand classification

### Phase 5: Plugin — Copy & Adapt Loot Files
1. Copy VTankLootParser.cs -> update namespace, replace Decal types with local types
2. Copy LootManager.cs -> apply all substitutions from table above
3. Simplify to single loot evaluation path (VTankLootParser only)
4. Always-RequestId for NeedsID

### Phase 6: Plugin — Integration
1. Wire LootManager into NexAiPlugin.cs
2. Add settings to LegacyUiSettings.cs
3. Add chat commands for profile loading
